### 结论摘要
1. **相变前提是完美验证**：7B群体智慧的“相变”极度依赖底层白盒的**绝对确定性校验**（P vs NP）。若缺乏确定性验证，几百个7B只会发生“幻觉共振”，放大噪声。
2. **“大Planner+小Worker”是最优解**：Planner负责将复杂问题“降维”和“切片”（确保方向正确），7B Worker负责在狭窄且明确的搜索空间内“穷举”（暴力破解）。
3. **硬件真相是“大Batch”而非“多实例”**：在单/双GPU预算下，上百个Worker不是上百个模型实例，而是一个7B模型通过 vLLM/SGLang 开启极高并发 Batch Size 与高 Temperature（采样多样性）。
4. **验证Zero Error无需1M步**：只需在一个深度为50-100步的马尔可夫决策过程（MDP）中，证明系统具备**100%的错误拦截与回滚率**，即可在数学上等效推导到1M步。
5. **架构必须严格解耦**：顶层只做状态路由，中层只负责猜想（生成），底层只负责证伪（执行）。越界必死。
6. **先升Planner，后扩Worker**：如果Planner切分的子任务超出了7B的能力上限（概率$P=0$），Worker数量$K$放大一万倍成功率依然是0。
7. **拒绝LLM-as-a-Judge**：系统流转的决定权必须在⚪顶层或底层手中，决不能让Planner或Worker自己投票决定代码是否真正跑通。
8. **时间重于空间**：长程任务的核心挑战不是上下文窗口有多大，而是状态树（State Tree）有多深，防退化机制是唯一生命线。

### 详细方案

#### 1) 概率与系统工程视角的“百个7B + 大Planner”
*   **有意义的条件**：
    *   任务符合 **验证容易，生成困难 (P vs NP)** 的特征（如：数学证明、代码编译、正则匹配、确定性API调用）。
    *   大Planner具备将任务拆解为“7B能在1000个Token内解决”的原子子任务的能力。
    *   **数学逻辑**：设7B单次解决原子任务的概率为 $p=0.05$。并发 $K=100$ 次，只要底层白盒能100%正确筛选，单步成功率 $P = 1 - (1 - 0.05)^{100} = 99.4\%$。连续1000步的成功率为 $0.994^{1000} \approx 0.24\%$（仍然不够，需引入重试与回滚），但如果 $p=0.2$，$P \approx 99.9999999\%$，即可逼近Zero Error。
*   **放大噪声的条件**：
    *   开放性文本生成、主观逻辑推断、或者无法用代码/工具验证的场景。此时只能依赖 LLM 投票，而同质化 7B 模型的系统性偏差会导致它们大概率在同一个错误上达成共识。

#### 2) 最小可验证实验 (MVP) - 3阶段测试
**目标**：不测1M步，测能否在确定性边界内触发“相变”。
*   **阶段一：单步穷举相变测试 (N=1)**
    *   **输入**：困难级别的合成代码题（如 LeetCode Hard 的变种），自带5个隐藏测试用例。
    *   **输出**：通过所有测试用例的函数。
    *   **验证门槛**：绘制成功率曲线。设 Worker 数量 $K \in [1, 5, 10, 50, 100, 200]$。如果成功率在某个 $K$ 值（如 $K=40$）从 10% 陡增到 90%+，说明**相变点存在**。
*   **阶段二：深度为10的马尔可夫链 (N=10)**
    *   **输入**：需要按顺序调用10次不同API（带前后依赖、需解析JSON）的微型迷宫任务。
    *   **输出**：最终状态的正确Flag。
    *   **验证门槛**：引入大Planner做步骤规划。指标为“错误拦截率”。要求：Worker发生错误生成的次数 $> 100$ 次，但**错误穿透率必须为 0%**（全被底层拦截并触发重试），最终任务成功率 $> 80\%$。
*   **阶段三：回滚与死锁恢复测试 (N=50)**
    *   **输入**：带“陷阱路径”的系统重构任务（如果做错第5步，到第15步才会报错）。
    *   **输出**：重构完成的代码库。
    *   **验证门槛**：系统必须能沿着状态树（MCTS/DFS）向上回滚。要求：触发至少3次跨步回滚后，依然能到达终点。

#### 3) 推荐拓扑结构 (⚪⚫⚪ 反奥利奥)
*   **⚪ 顶层白盒 (Global State & Consensus)**
    *   **设计**：严格的有限状态机 (FSM) 或 蒙特卡洛树搜索 (MCTS) 节点管理器。
    *   **核心功能**：维护当前全局上下文（State），控制回滚（Rollback）与分支剪枝。
    *   **熔断机制**：同一个节点的重试次数达到 `max_retries` 即标记为死胡同（Dead End），强制回退父节点。
*   **⚫ 中层黑盒 (Generation & Decomposition)**
    *   **分工**：
        *   **Planner (大模型)**：只在状态转移（Node Transition）时触发。输入全局历史，输出“下一步的3个可能方向/约束条件”。不写具体代码。
        *   **Workers (7B模型群)**：高频无状态触发。接收 Planner 的一个方向指令，带上极高 Temperature，疯狂生成代码/动作。
*   **⚪ 底层白盒 (Ground Truth & Execution)**
    *   **设计**：沙箱环境、AST解析器、Linter、编译脚本。
    *   **核心功能**：执行 Worker 的动作，返回绝对客观的 `Pass/Fail` 信号与 `Error Trace`。决不涉及任何神经计算。

#### 4) 预算敏感条件下的并发策略 (1~2台 Spot GPU)
*   **1台 GPU (如单卡 24G/40G/80G)**:
    *   **策略**：放弃分离的大 Planner。采用 14B~32B 级别的单一模型（如 Qwen2.5-32B AWQ/GPTQ 量化版）。
    *   **排布**：利用 vLLM 部署。以 `temperature=0.1` 跑1次 Planner Prompt，然后立刻以 `temperature=0.8` 跑 `n=100`（Batch Size）的 Worker 采样。
*   **2台 GPU**:
    *   **策略**：物理隔离，流水线作业。
    *   **GPU 1 (Planner)**：部署 70B 模型（INT4/8量化），低吞吐，高智商，负责规划与分析底层报错日志。
    *   **GPU 2 (Workers)**：部署 7B 模型（FP16），使用 vLLM 极限拉满 Batch Size，开启 Prefix Caching（因为几百个 Worker 的 System Prompt 和任务前缀完全一致），只并发生成最后几百个 Token。
*   **优先级**：**绝对是先升 Planner，后扩 Worker。** Worker 是瞎子摸象，如果 Planner 给的池子里没有大象，摸一万次也是徒劳。

#### 5) “不要做”清单 (避免伪进展)
1. **不要用 LLM 做最终代码/状态的正确性评估**（LLM可以给思路，但判卷必须是编译器）。
2. **不要让 Worker 维护长会话历史**。Worker 必须是无状态的单次调用 (Stateless API)，状态由顶层白盒拼接。
3. **不要使用外部网络/不可控的第三方API**作为底层工具，网络波动会污染你的“零错误”理论验证。
4. **不要追求 Worker 之间的直接通信**（多智能体对话是低效的），它们只能通过顶层状态机间接影响。
5. **不要为 Worker 设过低的 Temperature**。批量采样时 $T < 0.5$ 会导致生成高度同质化，失去并发试错的意义。
6. **不要在没有实现“沙箱秒级回滚”前跑长程任务**。底层脏数据累积会瞬间摧毁系统。
7. **不要指望 7B 模型自己做长远规划**。只要上下文超过 4K 且涉及多步骤，7B 必然断片。
8. **不要把解析器 (Parser) 写的太宽容**。Worker 输出格式错误直接判 Fail 并惩罚，不要试图用正则去勉强挽救，否则注入漏洞无穷无尽。

#### 6) 7天执行计划 (MVP 验证路线)
*   **Day 1**: 部署 vLLM 环境，跑通单卡 7B 模型 `n=100` 批量并发采样，验证 Prefix Caching 效率与 Token/s 吞吐极值。
*   **Day 2**: 纯 Python 编写底层白盒（沙箱执行器，确定性评分器）与合成测试题（Stage 1）。
*   **Day 3**: 将中层 Worker 接入底层白盒，跑通并绘制阶段一的 $K$ 值与成功率“相变曲线”。
*   **Day 4**: 引入 Planner 模型，纯 Python 编写顶层白盒（树搜索状态机，支持 BFS）。
*   **Day 5**: 组装完整 ⚪⚫⚪ 架构，运行 Stage 2 (深度10的任务)，调试报错解析与 Worker 重试循环。
*   **Day 6**: 注入死胡同陷阱（Stage 3），验证顶层状态机的跨节点回滚能力与 Zero Error 绝对拦截率。
*   **Day 7**: 分析实验数据，盘点吞吐瓶颈与 API 成本对应关系，出具《规模化可行性审计报告》。

| 参数项 | 建议值 | 工程解释 |
| :--- | :--- | :--- |
